# 过往采集脚本复用评估

## 总体判断

过去的脚本证明了部分淘宝页面读取、店铺主页识别和双路线思路可以工作，但它们是阶段性原型。新的测试调度器应读取 `data/fixtures/业务知识库测试集/price_specialist_test.sqlite3`，不能继续把旧 CSV 当成正式数据库。

## 主脚本

| 文件 | 判断 | 可以复用 | 不能原样保留的部分 |
|---|---|---|---|
| `run_taobao.py` | 保留为淘宝双路线原型，不直接作为正式调度入口 | OpenCLI命令调用、JSON解析、药品和规格核对、Search候选评分、按店保存结果 | 硬编码路径和CSV、异常后继续尝试、随机模拟人工行为、直接覆盖结果CSV、缺少测试库和统一队列接入 |
| `run_jd.py` | 保留为京东限流诊断和双路线参考，禁止原样重跑 | 详情与搜索分流、缓存去重、字段核对、失败证据结构 | 当前版本未可靠识别限流/验证码/登录异常，不能保证暂停京东平台 |

## 辅助脚本

| 文件 | 判断 | 后续处理 |
|---|---|---|
| `get_shop_home.py` | 店铺真实主页链接、店铺ID提取方法有价值 | 将核心逻辑抽成淘宝适配器的 `resolve_store_identity`，去掉固定店铺和固定输出路径 |
| `generate_summary.py` | 汇总思路可用，当前脚本只适合查看历史批次 | 后续改为从测试库和运行结果表动态生成，不再写死“20家、3家”等结论 |
| `verify_3stores.py` | 检查点有价值，脚本本身不应继续直接运行 | 把Search独立成功、SKU、主页真实性、可疑价格四项迁成离线自动测试后归档 |
| `verify_remaining.py` | 与前一个验证脚本高度重复 | 检查项迁移完成后可以删除 |

## 输入与输出文件

| 文件类型 | 判断 |
|---|---|
| `taobao_store_tasks.csv`、`jd_store_tasks.csv` | 保留为历史小批任务样本，不再作为新调度器主输入 |
| `historical_product_clues.csv` | 保留为历史链接证据，可用于对照测试库抽样结果 |
| `shop_home_verified.csv` | 保留，3家已人工验证店铺可作为店铺身份回归基线 |
| 淘宝映射、结果、未解决项和日志 | 保留为历史运行样本，不覆盖、不继续追加正式业务结果 |
| 京东结果、总结、执行日志和限流截图 | 整包保留，作为“平台异常后暂停两条路线”的回归样本；它们并非全部空数据 |

## 底层模块

| 模块 | 判断 |
|---|---|
| `pricing.py` | 价格解析和计算逻辑可直接复用 |
| `catalog.py` | 药品识别与规格归一化可复用，但以后应由知识库药品主数据驱动 |
| `smoke_plan.py` | 店铺名称归一化函数可复用 |
| `enums.py`、`config.py` | 状态和配置结构可复用 |
| `collector.py` | 结构可复用，但必须先恢复验证码、限流和登录异常识别，否则不能上线 |

## 保留层级

1. **进入新主流程：** 解析纯函数、规格归一化、候选评分、店铺主页提取、价格计算。
2. **历史原型保留：** `run_taobao.py`、`run_jd.py`、`get_shop_home.py`、`generate_summary.py`。
3. **迁成自动测试后归档：** 两个 `verify_*.py`。
4. **历史证据保留：** 旧CSV、汇总报告、限流日志和截图。
5. **可以直接清理：** `__pycache__/`、`.pyc`、不存在的旧标准输出日志，以及未被Excel占用的 `~$` 临时锁文件。

## 新调度器的接入原则

- 路线一读取测试库的 `monitor_task_master`。
- 路线二独立读取 `search_seed_tasks`，不能依赖已有历史线索才能搜索。
- 采集结果写入独立 `run_id`，不能覆盖测试基准和历史输出。
- 验证码、限流、强制登录或账号风险出现时，暂停该平台两条路线；其他平台继续。
- 搜索列表价只作为候选，详情确认后才生成正式价格。
