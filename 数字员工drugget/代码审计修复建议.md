# 二、多 Agent 执行方式

主 Agent 必须将任务拆分并派遣给多个 sub-agent 并行处理。

建议至少拆分为以下 5 个工作流。

## Sub-agent A：环境隔离与配置系统

负责：

* `config.py`
* `.env`
* `.env.test`
* `.env.prod`
* 数据库 URL
* evidence/output 目录隔离
* 测试模式安全检查

目标：

* 修复当前 `_load_dotenv()` 使用 `os.environ.setdefault()` 导致后加载配置无法覆盖前置配置的问题。
* 避免在 GUI 多次运行时污染全局 `os.environ`。
* 保证同一进程中先运行正式模式再运行测试模式，或者先测试再正式，配置都不会串线。
* 测试模式必须使用测试数据库、测试 evidence 目录和测试输出目录。
* 正式模式必须使用正式数据库和正式目录。
* GUI 开始运行前必须展示脱敏后的实际运行配置。

## Sub-agent B：任务生命周期与取消机制

* `test_runner.py`
* `orchestrator.py`
* `services.py`
* 任务状态
* run 状态
* 取消逻辑

# 三、必须修复的问题

## 1. 测试和正式环境隔离失效

当前 `Settings.from_env()` 会按顺序读取：

* `.env`
* `.env.test` 或 `.env.prod`

但 `_load_dotenv()` 使用：

```python
os.environ.setdefault(key, value)
```

导致 `.env.test` 和 `.env.prod` 无法覆盖 `.env` 中已经设置的值。

同时，配置被写入进程级 `os.environ`，同一个 GUI 进程多次执行时会互相污染。

必须重构为无副作用的配置加载。

推荐设计：

```python
Settings.load(
    project_dir,
    mode="test",
    overrides={...},
)
```

配置优先级必须明确为：

```text
显式调用参数
> 当前模式配置文件（.env.test 或 .env.prod）
> 公共 .env
> 系统环境变量或项目约定的明确优先级
> 默认值
```

无论采取哪一种优先级，必须：

* 明确定义。
* 自动化测试覆盖。
* 不依赖之前运行遗留在 `os.environ` 中的值。
* 不允许测试模式意外连接正式数据库。

建议增加安全保护：

```python
settings.validate_runtime_mode()
```

测试模式启动时应验证：

* database URL 指向测试库；
* evidence 目录属于测试目录；
* output 目录属于测试目录；
* 不满足条件时拒绝启动。

GUI 启动时显示：

```text
运行模式：TEST
数据库：.../price_specialist_test.sqlite3
证据目录：.../artifacts/test/evidence
结果目录：.../artifacts/test/runs
```

数据库密码等敏感信息必须脱敏。

## 2. 停止按钮没有真正取消执行

当前：

```python
TestWorker.cancel()
```

只设置 `_cancel_flag`，但执行链路没有读取它。

必须实现 cooperative cancellation。

建议新增统一接口：

```python
class CancellationToken:
    def cancel(self) -> None: ...
    def is_cancelled(self) -> bool: ...
    def raise_if_cancelled(self) -> None: ...
```

把 cancellation token 传入：

* `TestWorker`
* `BatchOrchestrator`
* 必要的 collector 包装层

必须在以下位置检查取消状态：

1. 创建批次后。
2. 每个平台开始前。
3. 每个 stage 开始前。
4. 每次 lease 新任务前。
5. 当前任务执行结束后。
6. sleep 前后。
7. 创建详情任务前。
8. 导出前。

取消后的预期行为：

* 当前正在执行的不可中断外部操作可安全结束，但不得 lease 下一任务。
* pending/leased 但未开始的任务标记为 `cancelled`。
* run.status 标记为 `cancelled`。
* run.finished_at 正确写入。
* GUI 等到 worker 确认退出后再显示“已停止”。
* 不得发送 `done/success`。
* 可以导出取消现场，但必须标记为取消结果，而不是成功结果。
* 停止后不能丢失后台线程引用，直到线程真正结束。

如果 OpenCLI 子进程可以安全终止，应提供受控终止能力；如果不能，必须明确界面提示“正在结束当前操作”。

## 3. 执行完成与采集成功混为一谈

当前 `TestWorker` 在 `execute_all()` 正常返回后无条件发送：

```python
status="success"
message="采集完成"
```

即使所有平台：

* 登录失败；
* 验证码阻塞；
* 搜索零命中；
* 所有任务失败；
* 没有任何正式价格结果；

GUI 仍可能显示成功。

必须拆分至少三个维度：

```text
execution_status
business_status
run_status
```

推荐：

### execution_status

```text
queued
running
finished
cancelled
crashed
```

表示程序执行生命周期。

### business_status

```text
success
partial_success
no_result
failed
blocked
cancelled
```

表示业务采集结果。

### task_status

```text
pending
leased
running
succeeded
not_found
failed
human_required
blocked
skipped
cancelled
```

不要再把 `not_found` 简单当作系统异常，但也不能算作成功抓取。

最终批次摘要应包含：

```json
{
  "execution_status": "finished",
  "business_status": "partial_success",
  "task_counts": {
    "total": 20,
    "succeeded": 6,
    "not_found": 8,
    "failed": 3,
    "blocked": 2,
    "cancelled": 1
  },
  "drug_counts": {
    "total": 10,
    "captured": 4,
    "not_captured": 5,
    "partial": 1
  },
  "formal_price_count": 3
}
```

GUI 颜色建议：

* 绿色：至少达到用户定义的成功标准。
* 黄色：部分成功、零命中或存在失败。
* 红色：全部失败、阻塞或异常。
* 灰色：取消。

## 5. 编排器进度回调

`BatchOrchestrator` 应支持事件 sink 或 callback：

```python
class RunEventSink(Protocol):
    def emit(self, event: RunEvent) -> None:
        ...
```

TestWorker 传入一个组合 sink：

```python
CompositeEventSink(
    JsonlEventSink(...),
    QueueEventSink(gui_queue),
    DatabaseEventSink(session),
)
```

编排器在任务生命周期关键节点发送事件。

不要让 TestWorker 在运行结束后猜测执行过程。

## 6. 定期状态快照

除事件流外，建议每隔固定时间或固定事件数输出：

```text

task_status.csv
```

### `run_status.json`

### `drug_status.csv`

### `task_status.csv`

快照必须使用原子写入：

```text

```

避免 GUI 或用户读取到半写文件。

# 六、测试要求

所有修复必须增加测试。

至少增加以下测试。

## 配置隔离测试

1. `.env` 指向正式库，`.env.test` 指向测试库，测试模式必须使用测试库。
2. `.env` 指向测试库，`.env.prod` 指向正式库，正式模式必须使用正式库。
3. 同一进程先运行 test，再运行 prod，不能串库。
4. 同一进程先运行 prod，再运行 test，不能串库。
5. 测试模式连接正式数据库时必须拒绝启动。
6. test/prod 的 evidence 和 output 目录必须不同。

## 取消测试

1. 取消后不再 lease 新任务。
2. 取消后 pending 任务进入 cancelled。
3. run.status 为 cancelled。
4. 不发送 done/success。
5. GUI 不在 worker 仍运行时清空 worker 引用。
6. 取消后不继续导出成功状态。

## 状态语义测试

1. 全部任务成功：business_status=success。
2. 部分成功：partial_success。
3. 全部搜索零命中：no_result。
4. 全部任务失败：failed。
5. 平台健康检查失败：blocked。
6. 用户取消：cancelled。
7. failed 任务存在时，不得显示纯成功。

## 店铺规划测试

1. 默认不会创建所有药品 × 所有店铺的笛卡尔积。
2. 只创建与药品相关的店铺任务。
3. 淘宝无 shop_home_url 时不入队。
4. 药师帮无可信 provider_id 时不直接入队 provider profile 采集。
5. 手工店铺选择只创建选定店铺任务。
6. 空店铺集合必须产生明确状态，而不是静默成功。

## 参数测试

1. search_limit 传递到适配器。
2. candidate_limit 限制候选保存数量。
3. inspect_limit 限制详情任务数量。
4. 无效参数被 GUI 和 Runner 拒绝。
5. 不允许存在只写入 payload 但无人读取的配置。

## 可观测性测试

1. 每个任务至少产生 started 和终态事件。
2. 搜索有结果产生 search_hits_received。
3. 搜索无结果产生 search_no_hits。
4. 详情成功产生 detail_succeeded。
5. 正式价格确认产生 formal_price_confirmed。
6. 每个事件包含 run_id、task_id、药品和平台上下文。
7. 药品状态聚合结果正确。
8. GUI queue 能收到真实 completed/failed/total 计数。
9. JSONL 可以按 run_id 重放恢复药品状态。
10. 状态快照文件使用原子写入。

## 端到端测试

构建完全离线的 fake collector：

* 一个药品搜索成功并详情成功；
* 一个药品搜索零命中；
* 一个药品详情解析失败；
* 一个平台登录阻塞；
* 一个运行中途取消。

完整执行：

```text
TestRunConfig
→ TestWorker
→ Task Planner
→ Task Queue
→ BatchOrchestrator
→ Fake Collector
→ Database
→ Run Events
→ Drug Status
→ Export
```

验证：

* GUI 事件。
* 数据库状态。
* 药品级结果。
* run 最终状态。
* CSV。
* manifest。
* 没有 fixture 输入污染。
* 没有正式数据库访问。

# 七、兼容性要求

1. 不得复制一份新的编排核心到 test runner。
2. fixture runner 和 GUI runner 必须共享正式 `BatchOrchestrator`。
3. 测试专用行为通过依赖注入、配置和 fake collector 实现。
4. 不允许在正式核心中写死 fixture 路径。
5. 不允许为了让测试通过而降低真实业务校验。
6. 不允许吞异常后继续显示成功。
7. 不允许以自由文本日志作为唯一状态来源。
8. 不允许修改 live 平台数据。
9. 不允许在自动化测试中访问真实淘宝或药师帮。
10. 保持现有 CLI 的合理兼容性；如必须变更接口，提供清晰迁移说明。

# 九、验收标准

只有同时满足以下条件才能认为修复完成。

1. GUI 勾选测试数据库时，能够证明连接的是测试数据库。
2. 同一 GUI 进程多次切换 test/prod 不会串库。
3. 点击停止后，后台不再继续 lease 或执行新任务。
4. GUI 不会把登录失败、全部零命中或全部失败显示为绿色成功。
5. GUI 可以实时显示每个药品的状态。
6. 用户可以明确看到哪些药品抓取成功。
7. 用户可以明确看到哪些药品未抓取成功。
8. 用户可以看到每个失败药品的失败平台、阶段和原因。
9. 进度条显示真实任务总数、完成数和失败数。
10. 店铺搜索不会默认生成无边界笛卡尔积。
11. 无可执行店铺时有明确提示。
12. `search_limit`、`candidate_limit`、`inspect_limit` 全部真实生效。
13. GUI 搜索过滤不会丢失药品选择。
14. 输出目录只包含本次运行相关数据。
15. manifest 能说明本次运行实际使用了什么配置和输入。
16. fixture runner 和 GUI runner 不互相污染。
17. 所有新增测试通过。
18. 不访问真实平台的离线端到端测试完整通过。
19. 原有测试无回归。
20. 主 Agent 提供逐文件修改说明、测试结果和剩余风险。
